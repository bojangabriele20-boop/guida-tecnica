📦 Formati dati
.json
JSON
JavaScript Object Notation
›

Cos'è: formato testuale per rappresentare dati strutturati come coppie chiave-valore, liste e oggetti annidati. È lo standard del web moderno.

Estensione
.json
Leggibile da umani
✅ Sì
Supporto linguaggi
Tutti
Velocità parsing
⚡ Molto veloce
Esempio
{
  "nome": "Gabriele",
  "età": 16,
  "attivo": true,
  "interessi": ["fitness", "coding"],
  "indirizzo": {
    "città": "Milano"
  }
}
API REST configurazioni database leggeri risposta server package.json
Regola base: tutto tra { } è un oggetto, tra [ ] è una lista. Le chiavi vanno sempre tra virgolette doppie.
.csv
CSV
Comma-Separated Values
›

Cos'è: formato tabellare semplicissimo dove ogni riga è un record e le colonne sono separate da virgole (o punto e virgola). È il formato universale per dati a tabella.

Estensione
.csv
Struttura
Tabellare (righe/colonne)
Aperto con
Excel, Python, qualsiasi cosa
Dimensioni
Molto leggero
Esempio
nome,età,città
Gabriele,16,Milano
Marco,17,Roma
Sara,16,Napoli
database export fogli Excel log dati machine learning datasets
Attenzione: se un valore contiene una virgola, va messo tra virgolette: "Rossi, Marco". In Italia spesso il separatore è il punto e virgola ;.
.xml
XML
eXtensible Markup Language
›

Cos'è: formato basato su tag annidati (simile all'HTML). Era lo standard per lo scambio dati prima del JSON. Oggi è ancora usato in sistemi aziendali, feed RSS e alcune API vecchie.

Estensione
.xml
Verbosità
🔴 Alta (molto testo)
Usato oggi
Sistemi legacy, RSS
Alternativa moderna
JSON (più snello)
Esempio (stesso dato del JSON sopra)
<utente>
  <nome>Gabriele</nome>
  <età>16</età>
  <interessi>
    <item>fitness</item>
    <item>coding</item>
  </interessi>
</utente>
feed RSS SOAP API Word/Excel interni configurazioni Java
JSON vs XML: JSON è più corto e più veloce da leggere. XML è più verboso ma supporta attributi nei tag e validazione schema. Preferisci sempre JSON se puoi scegliere.
⚙️ File script & eseguibili
.py
Python Script
Il linguaggio dell'automazione
›

Cos'è: file di testo contenente codice Python. È il linguaggio più usato per automazione, data analysis, AI, bot e script. Viene eseguito dall'interprete Python.

Esempio — script automatico per rinominare file
# rinomina tutti i .txt in una cartella
import os

for file in os.listdir("."):
    if file.endswith(".txt"):
        nuovo = file.replace(".txt", "_backup.txt")
        os.rename(file, nuovo)
        print(f"Rinominato: {file} → {nuovo}")
automazione web scraping bot Telegram AI/ML API calls pipeline dati
Come si esegue: da terminale con python nomefile.py oppure su Android con Pydroid3.
.sh
Bash / Shell Script
Automazione su Linux & Mac
›

Cos'è: sequenza di comandi da terminale salvata in un file. Permette di automatizzare operazioni di sistema: spostare file, eseguire programmi, fare backup, ecc.

Esempio — backup automatico
#!/bin/bash
# crea un backup della cartella Documenti

DATA=$(date +%Y-%m-%d)
zip -r "backup_$DATA.zip" ~/Documenti
echo "Backup creato: backup_$DATA.zip"
Linux/Mac backup CI/CD deploy server cron jobs
Prima riga obbligatoria: #!/bin/bash dice al sistema quale interprete usare. Per eseguire: chmod +x script.sh poi ./script.sh.
.bat
Batch File
Automazione su Windows
›

Cos'è: equivalente Windows del file .sh. Contiene comandi del prompt dei comandi (cmd.exe). Si esegue con doppio click o dal terminale.

Esempio — apri 3 siti e poi spegni il PC
@echo off
start chrome https://youtube.com
start chrome https://github.com
echo Siti aperti!
REM shutdown /s /t 60  (spegni tra 60 sec)
solo Windows automazione locale avvio programmi script di setup
Alternativa moderna su Windows: PowerShell (.ps1) è molto più potente. Il .bat è rimasto per compatibilità con sistemi vecchi.
🔧 File di configurazione
.yaml
YAML
YAML Ain't Markup Language
›

Cos'è: formato di configurazione leggibile progettato per essere scritto da umani. Usa l'indentazione (spazi) per la struttura, senza parentesi o virgolette ovunque. È lo standard per Docker, GitHub Actions, Kubernetes.

Estensione
.yaml o .yml
Sensibile a
⚠️ Indentazione (spazi!)
Esempio — configurazione app
server:
  host: localhost
  port: 8080
  debug: true

database:
  nome: mydb
  utenti_max: 100

features:
  - login
  - notifiche
  - export_csv
Docker Compose GitHub Actions Kubernetes configurazioni app
Errore comune: usare TAB invece degli spazi. YAML accetta solo spazi per l'indentazione. Un TAB rompe tutto senza dare errori chiari.
.env
Environment File
Variabili d'ambiente & segreti
›

Cos'è: file di testo semplice che contiene variabili d'ambiente. Serve a tenere separati i dati sensibili (password, API key, URL) dal codice. Non viene mai caricato su GitHub.

Esempio — .env di un progetto Python
# NON caricare mai questo file su GitHub!
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxx
DATABASE_URL=postgresql://user:pass@localhost/mydb
DEBUG=true
MAX_REQUESTS=100
Come si usa in Python
from dotenv import load_dotenv
import os

load_dotenv()
api_key = os.getenv("ANTHROPIC_API_KEY")
print(api_key)  # sk-ant-xxxx...
API keys password DB configurazione ambienti deploy
Regola d'oro: aggiungi sempre .env al file .gitignore per non caricarlo accidentalmente su GitHub e non esporre le tue credenziali.

💡 Concetti fondamentali
API
API
Application Programming Interface
›

Cos'è: un'interfaccia che permette a due programmi di comunicare tra loro. È come un cameriere: tu fai una richiesta, lui va in cucina (il server) e ti porta la risposta.

Analogia visiva
TU (client)  →  richiesta HTTP  →  SERVER
TU (client)  ←  risposta JSON   ←  SERVER

Esempio reale:
[app meteo] → "dammi temp. Roma" → [server meteo]
[app meteo] ← { "temp": 22, "cielo": "sereno" } ←
Chiamata API reale in Python
import requests

risposta = requests.get(
    "https://api.openweathermap.org/data/2.5/weather",
    params={"q": "Roma", "appid": "TUA_API_KEY"}
)

dati = risposta.json()  # converte in dizionario
print(dati["main"]["temp"])
Metodi principali
GET · POST · PUT · DELETE
Formato risposta
Quasi sempre JSON
Autenticazione
API Key o Token
Protocollo
HTTP/HTTPS
REST API Anthropic YouTube API meteo pagamenti social media
HTTP
Metodi HTTP
GET · POST · PUT · DELETE
›

Cos'è: il "verbo" che dice al server cosa vuoi fare con una risorsa. È lo stesso concetto delle operazioni CRUD sui database.

GET    → leggi    → "dammi lista utenti"
POST   → crea     → "crea nuovo utente"
PUT    → aggiorna → "modifica utente #5"
DELETE → elimina  → "cancella utente #5"

# Codici risposta comuni:
200 OK          → tutto ok
201 Created      → creato con successo
400 Bad Request  → errore nella tua richiesta
401 Unauthorized → non sei autenticato
404 Not Found    → risorsa non trovata
500 Server Error → errore lato server

PKG
Package Manager
pip · npm · apt
›

Cos'è: strumento che installa, aggiorna e gestisce librerie esterne. Ogni linguaggio ha il suo. Ti permette di usare codice scritto da altri senza reinventare la ruota.

# Python → pip
pip install requests          # installa
pip install requests==2.28.0  # versione specifica
pip uninstall requests        # rimuovi
pip list                      # mostra installati

# JavaScript → npm
npm install axios
npm run start

sudo apt install python3
requirements.txt — lista dipendenze Python

Buona pratica: salva sempre le dipendenze con pip freeze > requirements.txt così chiunque può ricreare il tuo ambiente con pip install -r requirements.txt.

🛠️ Strumenti fondamentali
GitHub
GitHub & Git
Controllo versione & collaborazione
›

Git è uno strumento locale che traccia la storia di tutte le modifiche al tuo codice. GitHub è il sito dove carichi quella storia online, per salvarla nel cloud, condividerla o collaborare con altri.

Pensa a Git come a un salvataggio multiplo nei videogiochi: ogni "commit" è un punto di salvataggio a cui puoi tornare. GitHub è il server cloud dove carichi quei salvataggi.

Git
Strumento locale sul tuo PC
GitHub
Sito cloud per i tuoi repo
Repository (repo)
Cartella tracciata da Git
Commit
Punto di salvataggio con messaggio
Flusso base — dalla prima volta al caricamento
# 1. Configura Git (solo la prima volta in assoluto)
git config --global user.name "Gabriele"
git config --global user.email "tua@email.com"

# 2. Inizializza un repo nella cartella del progetto
git init

# 3. Aggiungi i file al "palco" (staging area)
git add .                  # aggiunge tutto
git add main.py            # aggiunge solo un file

# 4. Crea un punto di salvataggio (commit)
git commit -m "Prima versione del progetto"

# 5. Collega al repo su GitHub (lo crei prima su github.com)
git remote add origin https://github.com/tuonome/repo.git

# 6. Carica su GitHub
git push -u origin main
Comandi quotidiani — il ciclo normale
# Controlla cosa è cambiato
git status

# Vedi le differenze riga per riga
git diff

# Salva le modifiche
git add .
git commit -m "Aggiunta funzione di login"

# Carica su GitHub
git push

# Scarica aggiornamenti da GitHub
git pull

# Vedi la storia dei commit
git log --oneline
Branch — lavorare in parallelo senza rompere nulla
# Un branch è una copia del progetto dove sperimenti
# senza toccare il codice principale (main)

git branch nuova-funzione       # crea branch
git checkout nuova-funzione     # spostati sul branch
# oppure in un comando solo:
git checkout -b nuova-funzione

# ... fai le modifiche e i commit ...

# Torna al main e unisci le modifiche
git checkout main
git merge nuova-funzione

# Vedi tutti i branch
git branch
Scenari di emergenza
# Annulla modifiche non ancora salvate
git checkout -- nomefile.py

# Torna all'ultimo commit (DISTRUGGE le modifiche non salvate)
git reset --hard HEAD

# Clona un repo esistente da GitHub sul tuo PC
git clone https://github.com/utente/repo.git

# Vedi la storia grafica dei branch
git log --oneline --graph --all
Struttura tipica di un progetto su GitHub
mio-progetto/
├── main.py           # codice principale
├── requirements.txt  # dipendenze Python
├── .env              # segreti (NON caricato)
├── .gitignore        # cosa ignorare
└── README.md         # descrizione del progetto
controllo versione backup codice collaborazione open source portfolio CI/CD
Regola d'oro: fai commit spesso con messaggi chiari ("Aggiunto login" non "fix"). Ogni commit è un punto di salvataggio: più ne hai, più facile è tornare indietro se qualcosa si rompe.
Docker
Docker
Contenitori per applicazioni
›

Cos'è: Docker crea dei contenitori — ambienti isolati e portatili che contengono tutto il necessario per eseguire un programma: il codice, le dipendenze, la versione di Python, le configurazioni. Funziona uguale su qualsiasi macchina.

Immagina una nave da carico: non importa cosa c'è dentro i container — si caricano e scaricano allo stesso modo ovunque. Il tuo codice è il contenuto, Docker è il container standardizzato.

Image
Blueprint del contenitore (ricetta)
Container
L'image in esecuzione (piatto pronto)
Dockerfile
File con le istruzioni per creare l'image
Docker Hub
GitHub delle image Docker pubbliche
Dockerfile — istruzioni per costruire il tuo ambiente
# Parti da un'image base (Python 3.11 già installato)
FROM python:3.11-slim

# Imposta la cartella di lavoro dentro il container
WORKDIR /app

# Copia i file delle dipendenze e installale
COPY requirements.txt .
RUN pip install -r requirements.txt

# Copia tutto il codice del progetto
COPY . .

# Comando che si esegue quando il container parte
CMD ["python", "main.py"]
Comandi Docker essenziali
# Costruisci l'image dal Dockerfile
docker build -t mio-progetto .

# Avvia un container dall'image
docker run mio-progetto

# Avvia in background (-d) e mappa porta 8080→80
docker run -d -p 8080:80 mio-progetto

# Vedi i container in esecuzione
docker ps

# Vedi tutti i container (anche fermati)
docker ps -a

# Ferma un container
docker stop ID_CONTAINER

# Entra dentro un container (come aprire un terminale)
docker exec -it ID_CONTAINER bash

# Scarica un'image pubblica (es. database Postgres)
docker pull postgres
Docker Compose — gestire più container insieme
# docker-compose.yml
# Esempio: app Python + database Postgres

version: "3.8"

services:
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql://user:pass@db/mydb
    depends_on:
      - db

  db:
    image: postgres:15
    environment:
      - POSTGRES_PASSWORD=pass
      - POSTGRES_DB=mydb
Comandi Docker Compose
# Avvia tutti i servizi definiti nel compose
docker compose up

# Avvia in background
docker compose up -d

# Ferma tutto
docker compose down

# Ricostruisce le image e riavvia
docker compose up --build
Confronto — senza vs con Docker
SENZA Docker:
"Sul mio PC funziona... sul tuo no"
→ versioni diverse di Python
→ dipendenze mancanti
→ configurazioni diverse del SO

CON Docker:
"Funziona sul mio → funziona ovunque"
→ tutto l'ambiente è dentro il container
→ chiunque esegue docker run ottiene lo stesso risultato
→ deploy su server = identico al tuo PC locale
deploy applicazioni ambienti isolati microservizi CI/CD server database locali
Quando usarlo: quando vuoi che il tuo progetto funzioni uguale su qualsiasi macchina, o quando devi avviare servizi complessi (database, API, frontend) tutti insieme con un comando solo.
Actions
GitHub Actions
Automazione integrata in GitHub
›

Cos'è: sistema di automazione integrato in GitHub. Ogni volta che fai un'azione sul repo (push, pull request, ecc.) puoi far partire automaticamente dei job: eseguire test, costruire il progetto, fare il deploy su un server.

I workflow si definiscono con file YAML dentro la cartella .github/workflows/.

Esempio — esegui i test ad ogni push
# .github/workflows/test.yml

name: Test automatici

on:
  push:              # si attiva ad ogni push
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest   # usa una VM Linux
    steps:
      - uses: actions/checkout@v3  # scarica il codice
      - name: Installa Python
        uses: actions/setup-python@v4
        with:
          python-version: "3.11"
      - name: Installa dipendenze
        run: pip install -r requirements.txt
      - name: Esegui i test
        run: python -m pytest
CI/CD test automatici deploy automatico build notifiche
CI/CD in breve: CI = Continuous Integration (testa il codice ad ogni modifica), CD = Continuous Deployment (distribuisce automaticamente se i test passano). GitHub Actions fa entrambe le cose gratis per repo pubblici.
